## 1. Overview

Next on Wembley is a small web app that recommends TV shows to a two-person household (**Corey and Jaimie**). It produces three ranked recommendation lists:

1. **Co-watch (Global)** — shows the algorithm predicts both Corey and Jaimie would enjoy together
2. **Corey's Picks** — personalized for Corey
3. **Jaimie's Picks** — personalized for Jaimie

Recommendations include both **continuations** of in-progress shows and **new shows**. Rankings are visible, each partner can vote Agree / Disagree / Maybe on every recommendation, and votes influence future ranking and are visible to the other partner. Recommendations are generated by an LLM (Claude API) reasoning over watch history, ratings, in-progress shows, active streaming subscriptions, Canadian streaming availability, TMDb community ratings, and episode counts.

## 2. Background & Motivation

Existing streaming-service recommendations are siloed per platform and biased toward each platform's catalog. A couple watching across Netflix, Disney+, Apple TV+, Crave, etc. has no single tool that:

- Tracks watch history across all services in one place
- Recommends jointly (for two people watching together) as well as individually
- Filters by what's actually available on their subscriptions in their country
- Distinguishes "what should we start next" from "what should we continue"
- Explains *why* a show is being recommended

Next on Wembley fills that gap for personal household use.

## 3. Goals & Non-Goals

### Goals
- Let Corey and Jaimie independently track shows they've watched, are watching, dropped, or want to watch
- Provide three LLM-generated ranked recommendation lists: Co-watch (Global), Corey's, Jaimie's
- Recommendations consider watch history, in-progress shows, active streaming subscriptions, ratings/reviews, Canadian streaming availability, and episode count
- Let each partner vote Agree / Disagree / Maybe on rankings, visible to the other partner, with votes influencing future rankings
- Let each partner rate watched shows Like / Dislike / Meh, feeding back into future recommendations
- Surface in-progress shows with how many episodes remain to watch
- Exclude already-watched shows from recommendations unless new episodes have been released
- Support filtering by platform, genre, watched/in-progress, and whose pick a recommendation is
- Allow search of shows by title

### Non-Goals (initial release)
- Movie recommendations
- Direct integration with streaming services for playback or watch-progress sync
- Social features beyond Corey + Jaimie (no friend graph, no public sharing)
- Mobile native apps (responsive web is enough)
- Public sign-ups or multi-tenant scaling
- Markets outside Canada (region is fixed to CA in v1)

## 4. Target Users

- **Primary:** Corey and Jaimie — a two-person household who watch TV together and separately, want shared and personalized recommendations, and consume content across multiple streaming platforms in Canada.
- Comfortable with web apps; not necessarily technical.
- Each user has their own watch history, ratings, votes, and platform subscriptions; some recommendations are co-watch.

## 5. User Stories

### Account & Setup
- *As a user,* I can log in to my own profile so my watch history and votes stay separate.
- *As a user,* I can configure which streaming platforms I subscribe to (Netflix, Disney+, Apple TV+, Crave, Prime Video, etc.) so recommendations only suggest things I can actually watch.

### Watch History
- *As a user,* I can search for a TV show by title.
- *As a user,* I can add a show to my list with a status (Want to Watch / Watching / Paused / Completed / Dropped) and note which season I'm on.
- *As a user,* I can see a list of shows I'm actively watching, including how many episodes remain.
- *As a user,* I can rate any watched show as Like / Dislike / Meh.
- *As a user,* I can edit or remove entries from my watch history.

### Show Details
- *As a user,* I can see each show's cover image, number of seasons and episodes, genre, TMDb rating, trailer link, and the streaming platforms it's available on in Canada.

### Recommendations
- *As a user,* I can see three ranked lists: Co-watch (Global), Corey's Picks, Jaimie's Picks.
- *As a user,* I can vote Agree / Disagree / Maybe on each recommendation; my vote influences ranking and is visible to my partner.
- *As a user,* I can see how my partner voted on the same recommendation.
- *As a user,* I can filter any recommendation list by platform, genre, watched, in-progress, or whose pick.
- *As a user,* I trust that I won't see shows I've already watched in recommendations — unless new episodes have been released, in which case they re-enter the pool and are ranked accordingly.

## 6. Functional Requirements

### 6.1 Authentication
- Two pre-provisioned accounts: **Corey** and **Jaimie**. No public sign-up.
- **Shared-secret passcode per user.** Each user has a static passcode stored as a hash server-side; entering it issues a session cookie. No email provider or token-table dependency.
- Sessions persist across browser restarts via an HTTP-only cookie.
- **Persistent identity indicator.** The app shell shows an always-visible identity chip displaying the current user's name and initial / avatar — on mobile, in the top header (adjacent to the bottom tab bar in §7.2); on desktop, in the top-right corner. Tapping the chip opens a switch dialog (passcode entry for the other user) so the household can quickly hand off a shared device without accidental cross-voting.

### 6.2 User Settings
- Each user maintains a list of **active streaming subscriptions** (e.g., Netflix, Disney+, Apple TV+, Crave, Prime Video). This list is editable at any time.
- **Effect on recommendations:** when a user adds or removes a subscription, the app auto-triggers a regeneration of all three rec lists for that user (and Co-watch, since combined availability shifts). See §6.4.7 Refresh Cadence for the stale-list behavior during regeneration.
- Region is fixed to **Canada** in v1 (used for streaming availability lookups).

### 6.3 Watch History
- Add a show by searching TMDb (title autocomplete, poster preview, year disambiguation).
- Each entry has: show, status, **current season** (for `Watching` and `Paused` statuses), user rating **Like / Dislike / Meh** (optional, settable on any status — useful as an early taste signal on shows still in progress), free-text notes (optional), date added/updated.
- Statuses: `Want to Watch`, `Watching`, `Paused`, `Completed`, `Dropped`.
  - **Paused:** the user has started the show but is on hold (drifted off, between seasons, lost the remote, etc.). Carries no negative signal to the LLM. Excluded from rec lists (no continuations surface) until the user moves it back to `Watching`.
  - **Dropped:** active rejection. Strong negative signal; excluded from all recommendations.
- Edit and delete entries.
- **Remove ≠ Dropped.** Each entry card carries a Remove button (with a confirm step) that deletes the `WatchEntry` outright. Remove is a **neutral** action — it carries no signal in either direction. Use it when the original add was accidental (a misclick on Want-to-Watch, an exploratory add). `Dropped` is the right status when the user actively rejects a show and wants future recs to treat it as a negative signal.
- **In-Progress view:** shows with status `Watching` displayed prominently.
  - For each entry: title, poster, `current_season`, and an indicator of how many episodes/seasons remain (derived from TMDb's total episode count vs. user's current season).
  - Each entry also displays the show's **TMDb production status verbatim** (e.g., "Returning Series", "Ended", "Canceled") accompanied by a small caveat note (e.g., "Per TMDb — may change") to acknowledge that this status is not a permanent claim: shows are renewed, revived, or cancelled dynamically.
  - If a show is no longer available free-with-subscription on any of the user's active subscriptions in Canada, the entry shows an **"Unavailable on your subscriptions"** badge. The entry stays in the In-Progress view, and the show still appears as a continuation in rec lists (with the same badge). Hiding a partway-watched show is more disruptive than honest — the user may have an alternative way to watch (other household, library, friend's account). See §6.4.5 for the new-show vs continuation distinction.
  - `Paused` entries are hidden from In-Progress by default but can be revealed via a "Show Paused" toggle.
  - Each entry has inline controls to advance progress without opening the edit form: a **+1 / -1 season** button pair (instantly updates `current_season`), and a **"Finished it"** shortcut that prompts for an optional Like / Dislike / Meh rating and moves the entry to `Completed`.
- **Search:** dedicated show search by title across the user's entries and TMDb.
- **Bulk import:** there is no bulk-import path in v1. The user populates their watch history by searching and adding shows one at a time. A CSV bulk-import flow is deferred — see §11 Future Considerations.

### 6.4 Recommendations

#### 6.4.1 Lists
- Three distinct ranked lists:
  1. **Co-watch (Global)** — shows predicted to satisfy both Corey and Jaimie based on combined watch history, ratings, and overlapping platforms. **Carries 25 picks** because it's the household's default browsing surface (and the landing tab on /recs).
  2. **Corey's Picks** — personalized for Corey. **10 picks** to match the PRD §10 vote-on-top-10 metric.
  3. **Jaimie's Picks** — personalized for Jaimie. **10 picks**, same rationale.
- Per-scope sizes flow through the candidate-count parameter in the user prompt (system prompt stays generic so prompt caching survives the per-call variation). Co-watch asks the LLM for ~32 raw candidates and trims to 25; personal lists ask for 16 and trim to 10.
- Each list also includes **continuations** — in-progress shows that have new episodes/seasons available — **interleaved by LLM rank** alongside new-show recommendations. A continuation can land anywhere in the list, including #1, if the LLM judges it the strongest pick. There is no separate "Continue Watching" header section in v1.
  - For individual Picks lists, a continuation is any of that user's `Watching` entries with aired-but-unwatched episodes.
  - For Co-watch, a continuation requires **both users to have `Watching` entries on the same show at the same `current_season`**. If one user has advanced ahead, the show is no longer surfaced as a Co-watch continuation until they sync up. No explicit "co-watch" flag is stored on `WatchEntry` — co-watch status is inferred from the intersection.

#### 6.4.2 Algorithm Inputs (no fixed priority among inputs unless noted)
- Previous watch history (titles, statuses, ratings, genres)
- Existing shows being watched (in-progress entries)
- Active streaming subscriptions for the relevant user(s)
- Reviews/ratings (TMDb community rating)
- Streaming availability — must be **free with subscription** and available in **Canada**
- Number of episodes — more episodes is mildly preferred, but **overall low priority**
- Optional **mood input** — a free-text field the user can fill in at refresh time (e.g., "something light tonight"). Passed to the LLM as additional context for that run only; not persisted as a taste signal.

#### 6.4.3 Ranking & Display
- The LLM assigns each recommendation a rank (1, 2, 3, …). Rank is **implicit from card position in the list** — the card UI does not display a "#3" label. Filters subset the list without reordering, so position always reflects LLM-assigned rank.
- **Rec card has two presentations:**
  - **Compact (default).** Cover image, title, top 2 platform chips + "+N more" overflow, your vote pills (Agree / Disagree / Maybe), partner-vote indicator (M4+), one-line LLM explanation (compact text, ≤100 characters), "Add to Want to Watch" button, and a "More" affordance to expand.
  - **Expanded (inline, on tap of "More").** Adds: full LLM explanation (long text, ≤300 characters), # seasons / # episodes, all genre tags, TMDb rating, trailer link, all platform chips.
- **Show Detail** (drawer or route per cluster K decision) is reached by tapping the poster or title, not the "More" affordance.
- **LLM explanation length is constrained** in the prompt: the LLM is asked to emit two versions per recommendation — a **short** (≤100 characters) and a **long** (≤300 characters). Both are stored; the compact card shows the short, the expanded card shows the long. This keeps card heights predictable and forces purposeful short copy rather than mid-sentence truncation.
- **Two distinct card action surfaces:**
  - **Vote pills** (Agree / Disagree / Maybe) — a ranking signal for the LLM.
  - **"Add to Want to Watch" button** — a commitment to actually watch this show. Adds a `WatchEntry` with status `want_to_watch` for the current user.
- Agreeing does **not** implicitly add the show to Want to Watch. The two intents are kept separate so the WTW list remains a curated commitment list, not an artifact of every Agree click.

#### 6.4.4 Voting
- For each recommendation, each user can vote **Agree / Disagree / Maybe**.
- **Votes are stored per `(user, show)`**, not per recommendation item. The same vote applies to the show across every list it appears on, and **persists across refreshes** — a new LLM run does not wipe prior votes.
- **List-owner enforcement.** A vote on a user-scoped tab is the *owner's* vote: only the user named in the tab (Corey on Corey's Picks, Jaimie on Jaimie's Picks) can cast or clear it. The partner viewing that tab sees the owner's vote rendered **read-only** — same pills, with the active selection visible and the buttons disabled (with a "Only the list owner can vote here" tooltip). This makes the partner's view a faithful preview of what the owner sees, while preventing accidental cross-user vote writes.
- **Co-watch voting** is per-viewer: each user casts their own vote on a Co-watch card. The owner-enforcement rule above does not apply because there is no single owner.
- **Partner visibility on Co-watch** (M4 Phase 25): each Co-watch RecCard surfaces the *other* household member's vote alongside the viewer's own pills as a small "{Partner}: Agree/Maybe/Disagree" chip. Display-only — the viewer can't mutate the partner's vote. Empty when the partner hasn't voted yet. User-scoped tabs (Corey's / Jaimie's Picks) don't get the indicator because that vote IS the owner's vote already.
- **Vote effects:**
  - **Agree:** boosts ranking signal for that user in the next LLM run.
  - **Disagree:** **hides the show** from that user's individual Picks list on display (immediate effect; positions renumber to close the gap), and is included in the recent-votes context passed to the LLM so future runs don't re-suggest it. Reversible via a one-click "Bring back" affordance in the **Buried disagrees** inspector at the bottom of the viewer's own Picks tab (M4 Phase 28) — clicking deletes the per-show vote, letting the show resurface in future refreshes (and, if it's still in the persisted current run, immediately reappear). The inspector only renders on the tab matching the viewer's username; Co-watch doesn't show it (no disagree filter there). In Co-watch, the show is *not* hidden in M3 — see the M4 split rule below.
  - **Maybe:** neutral-positive — counted as mildly interested but not actively boosted.
- **Re-casting votes:** votes are mutable. Each `(user, show)` pair has at most one stored vote; clicking a different pill overwrites the previous value, and clicking the already-active pill clears it (no vote). No vote history is kept in v1.
- **Voting on a continuation rec** (a show in the VIEWER's own `Watching` list — M4 Phase 27): clicking **Disagree** opens a Radix Dialog — *"Step back from "{title}"? You're currently Watching this show…"* — with three actions: **Move to Paused** (updates the WatchEntry status to `paused`, keeps the season pointer, records the Disagree), **Move to Dropped** (status → `dropped`, clears the season pointer, records the Disagree), or **Cancel** (no status change, no vote). The vote is only persisted alongside a status change — the two flips happen atomically inside `disagreeOnContinuationAction`. Skipped when the continuation is in the partner's history but not the viewer's (Co-watch case) — there's no WatchEntry to update, so the Disagree records normally. Agree and Maybe on continuations always behave normally with no prompt.
- **Votes do not re-rank the current list.** They are recorded and fed into the next LLM run when the user hits Refresh. This keeps the displayed rank consistent with the LLM's stated explanation for that run.
- **Co-watch split rule (M4 Phase 26 — implemented):** when partners have voted on the same show, the LLM applies a combination rule at rank time. Implementation lives entirely in the prompt: `generateRecommendations` intersects both users' recent votes by show title and passes the pairs as a "Vote combinations on shared shows" section in the co_watch user prompt; the system prompt's CO-WATCH SPLIT RULE block tells the LLM how to treat each combo:
  - Agree + Agree → strongly boost.
  - Agree + Maybe (either order) → boost.
  - Maybe + Maybe → neutral.
  - Agree + Disagree (split, either order) → **demote** (not exclude) — the Agreer's positive signal still earns a slot, but ranked below unanimous picks.
  - Disagree + Maybe (either order) → demote-heavy, usually exclude unless very strong fit.
  - Disagree + Disagree → exclude.

  This overrides the "don't recommend a Disagreed show" line for co_watch only — a single Disagree on a Co-watch candidate demotes rather than excludes. User-scoped lists keep the original hard-exclude semantics.
- A Disagree affects only the disagreeing user's individual Picks list. The partner can still see the show in their own Picks, and (subject to the split rule) in Co-watch.
- The LLM does **not** generalize from a Disagree to similar shows in v1 (see §11 Future Considerations).

#### 6.4.5 Exclusions & "Watched" Label
- Shows marked `Completed` or `Dropped` do **not** appear in recommendations and are labeled **Watched** in any view where they show up (e.g., filtered list, search results).
- Previously-Completed shows with no new episodes available are excluded from recommendations. (This rule applies only to entries with status `Completed` or `Dropped`; it does not block new-show candidates the user has never watched.)
- If new episodes/seasons become available for a previously-Completed show, it re-enters the recommendation pool and is ranked accordingly.
- **Subscription availability — new-show recs:** shows not available free-with-subscription in Canada on at least one of the relevant user's active subscriptions are **excluded** from new-show recs entirely. There's no commitment to break, and surfacing them would be unactionable.
- **Subscription availability — continuation recs:** shows the user has already started (`Watching` status) but that are no longer available on their active subscriptions **still appear** as continuations, labeled **"Unavailable on your subscriptions."** The user has an existing investment in the show and may have an alternative way to watch.
- **Continuation validation — aired-content check.** A continuation is only kept when the user has unwatched aired content remaining: either a later season has aired (per TMDb's listed per-season episode counts, which exclude announced-but-unaired seasons) or the user is mid-season. Shows where the user has finished the latest aired season are dropped from continuations even if TMDb's `totalSeasons` reflects a renewal — the next season exists on paper but has no episodes to watch yet. (Example: Severance S2 finished, S3 announced; not surfaced until S3 actually drops.)
- **Continuation validation — must be in history.** A rec marked as a continuation is dropped if the show is not on any relevant user's list (`Watching` or `Paused`) — the LLM occasionally invents continuations for shows nobody has started. Without this guard the show would bypass the subscription gate and surface as a stranded "continuation" with no underlying entry.
- **Per-list deduplication.** A single LLM response occasionally returns the same show twice (sometimes with near-identical explanations). The higher-ranked occurrence wins; subsequent duplicates are dropped before persistence.
- **Search behaves differently from recs.** Search returns matching TMDb results regardless of availability so the user can look up externally-sourced suggestions; titles not on the user's active subscriptions are labeled **Unavailable on your subscriptions** in search results. Users can still **add unavailable shows to Want to Watch** (useful for tracking shows they'd watch via a future subscription or another means); the resulting `WatchEntry` carries the same "Unavailable on your subscriptions" badge until availability returns.

#### 6.4.6 Filtering
- Each rec list can be filtered by:
  - **Platform** (subset of the user's subscribed platforms — non-subscribed platforms aren't shown as filter options since they're already excluded by the §6.4.5 hard filter at rec generation time).
  - **Genre** (derived from the genres present in the current list — empty when no items have parsed genres).
- Filters **subset** the visible list; they do not reorder or expand it. The list header shows `showing N of M` while any filter is active so the user knows what's been hidden.
- **Filters are URL-stateful.** Active filters serialize to query params (`?platform=netflix,crave&genre=Drama`) so the back button restores them and links can be shared with the filters applied. Multi-select (clicking a chip toggles it; the chip row + "Clear filters" affordance handle both directions).
- **Multi-select semantics:** an item matches if it has at least one hit in each active facet (intersection across facets, union within a facet). Example: `platform=netflix,crave` + `genre=Drama` matches Drama shows available on Netflix OR Crave.
- **Deferred filter facets:** Watched / In-Progress status filter (more relevant to Watch History views, which don't ship in M3); and Corey-recommended / Jaimie-recommended on Co-watch (depends on M4's whose-pick attribution).

#### 6.4.7 Refresh Cadence & Generation UX
- **Manual refresh** is the primary mechanism. A single **Refresh** button at the top of the Recommendations view regenerates all three lists (Co-watch, Corey, Jaimie) in parallel. There is no per-list refresh in v1 — this keeps Claude costs predictable, keeps the three lists consistent in age, and avoids refresh-fatigue.
- **Auto-refresh triggers in v1:**
  - **Subscription change** — when a user adds or removes an active subscription, the app auto-fires a regeneration of all three lists. This is the only auto-trigger in v1.
- **Generation runs in the background.** When Refresh is fired (manually or auto):
  - The currently displayed lists stay visible, **dimmed/grayed**, so the user can still browse and inspect old recs while waiting.
  - **Skeleton cards** with shimmer animation appear alongside (above the dimmed list, or as an "Updating…" header section) signaling that new content is incoming.
  - A small **"Refreshing recommendations…" pill** appears in the app header/nav so the user can navigate away (to History, Settings, etc.) and still know a refresh is in flight.
  - The user is **not blocked** — they can navigate freely, edit entries, add shows, change votes on the dimmed (still-current) cards.
  - When generation completes, the new lists fade in to replace the dimmed ones. If the user is off the rec page, the pill switches to **"Recommendations updated — view"** as a tappable affordance.
- **Stale-list behavior on subscription change** (the brief window between sub edit and regeneration completing): the displayed lists immediately apply a client-side filter:
  - New-show recs on cancelled subs are **hidden**.
  - Continuations on cancelled subs **stay visible**, badged "Unavailable on your subscriptions."
- **LLM response strategy:** v1 uses a **batch response** — single round-trip, full JSON returned, all cards rendered when the response lands. Response-streaming (cards-as-they-arrive) is deferred to a future iteration to reduce perceived latency (§11 Future Considerations).
- **No periodic auto-refresh in v1.** Votes, new ratings, and new Watching entries do not auto-trigger regeneration — they accumulate as input for the next manual Refresh.
- **Last-generated timestamp** is displayed at the top of each list (e.g., "Generated 4 hours ago") so the user always has visible freshness context.
- **Latency thresholds and error handling:**
  - 0–15s: normal "Refreshing…" state (dimmed stale list + skeleton cards + nav pill).
  - At 30s: an inline note appears near the skeletons — *"Taking longer than usual — the LLM is busy. Hang tight."* (Nav pill switches to "Still generating…")
  - At 60s: client-side timeout. Show an error card in place of the skeletons with **Retry** + **Dismiss** buttons. The dimmed stale list remains intact and visible so the user is never stranded with nothing. The underlying server action keeps running; a stale result that lands after the client timed out is dropped via an invocation token so it doesn't stomp the new state.
  - On transient errors (HTTP 5xx): the Anthropic SDK auto-retries **once** silently before surfacing the error card (`src/lib/anthropic.ts`). Rate-limit responses propagate directly without retry — they need user attention.
- **Failure messages surface the underlying cause** so the user (or operator) knows what to act on, rather than a generic "try again":
  - When every list shares the same error code, the message folds in the typed error from the SDK — e.g., *"All three lists failed to generate. Anthropic authentication failed — check ANTHROPIC_API_KEY."*
  - When TMDb resolution or the subscription gate drops every candidate (`no_valid_items`), the message reads *"…the recommendation service returned picks, but none could be matched against TMDb or your subscriptions. Try a different mood, or check your active subscriptions in /settings."*
  - On mixed error codes across the three lists, falls back to a generic *"N of 3 lists failed — try again in a moment."*
- **Mood input persists across failures.** Typed mood text stays in the input box across failed Refresh attempts; it is cleared only on a successful generation or by the user manually clearing it.

### 6.5 External Data
- TMDb is the source of truth for show metadata:
  - Title, poster, genres, season/episode counts, air dates, community rating (`vote_average`)
  - Production status (`status` field — e.g., "Returning Series", "Ended", "Canceled"; displayed verbatim with a "may change" caveat)
  - Trailers (`/tv/{id}/videos`)
  - Canadian streaming providers (`/tv/{id}/watch/providers` with region = `CA`, monetization = `flatrate`)
- **Per-season aired episode counts** are kept on `Show.seasonsJson` (`[{ seasonNumber, episodeCount }, …]`, season 0 specials filtered, zero-episode seasons dropped). This is the canonical source for "has new content aired" decisions — `totalSeasons` can include announced-but-unaired renewals and is not safe to use for continuation eligibility.
- **TMDb's CA provider data is uneven** — the same library surfaces under multiple `provider_id` values depending on access path (standalone app, Amazon Channel add-on, Apple TV Channel, ad-supported tier). We collapse every variant of each of the six supported platforms back to a single `PlatformKey` so subscription matching is robust even when only the channel-variant id is returned. Example: HBO content like *Succession* often returns only `2604` ("Crave Amazon Channel"), never `230` ("Crave") — both map to `crave`.
- Metadata is cached locally to avoid repeated API hits and to keep ratings stable between recommendation runs.
- Cache refresh policy:
  - Show metadata + ratings: weekly background refresh
  - Provider/availability data: every 14 days (changes more often)
  - In-progress shows: targeted refresh to detect new episodes

### 6.6 Show Detail

A single Show Detail surface is reached by tapping the poster or title from a rec card. Linking from watch history rows and In-Progress entries is **deferred** — those cards already carry inline edit affordances on the dashboard so the round-trip isn't necessary.

- **Content:**
  - Full TMDb metadata: title, poster, genres, season + episode counts (with aired-vs-announced split when they diverge), air dates, TMDb community rating, production status (with "may change" caveat per §6.3), trailer, **plot summary** (TMDb's `overview` field, displayed verbatim with a "Source: TMDb" attribution).
  - Canadian provider chips, showing every CA platform the show is on with a "Not on your subscriptions" badge when there's no overlap with the user's active subs.
  - The current user's relationship to the show: if a `WatchEntry` exists, status + `current_season` + rating displayed read-only with a pointer back to the dashboard for edits. Inline edit affordances (status change, +/- season, "Finished it" shortcut, rating) are **deferred** to a follow-up phase; the dashboard cards already handle these flows.
  - When entered from a recommendation context (`?recItem=N`): vote pills (the owner's vote per Phase 15.1 ownership rules — partner sees read-only on user-scoped lists) + "Add to Want to Watch" button, plus the LLM explanation in its long form (≤300 chars per §6.4.3). Voting and Add-to-WTW work the same as from the rec card.
- **Section order (top → bottom):** title + continuation badge → poster + metadata grid → Where to watch chips → (if rec context) "Why Claude picked this" with the long explanation + vote controls → "About the show" with TMDb's plot summary → "Your list" relationship summary. Claude's reasoning leads when present because the rec context is what brought the user here; TMDb's description sits beneath as objective background.
- **Routing pattern (current — Phase 20):** full page only at `/show/[tmdbId]` (optionally `?recItem=N` for rec context). In-app navigation pushes a new route; the list page yields to the detail page rather than overlaying a drawer.
- **Routing pattern (Phase 20b, deferred):** intercepting / parallel routes layer a drawer over the originating list when navigated in-app, preserving scroll position; the standalone full-page route still works for shared links and cold loads.
- **Close behavior:**
  - Full page: browser back, or the "Back to recs" affordance at the top of the page (always shown — useful when arriving from a shared link).
  - Drawer (Phase 20b): Esc, swipe-down (mobile), or X. Returns to the originating list at the prior scroll position.

### 6.7 UI States

#### 6.6.1 Empty States
Every screen with a list or data view has a tailored empty state with a clear next action — no dead ends.
- **Watch History (new user, zero entries):** "Add your first show" CTA with the search input prominently displayed.
- **In-Progress view (no `Watching` entries):** "Nothing in progress right now — see your Picks for what to start next." with a link to the Recommendations view.
- **Rec list (pre-first-generation, no signal at all):** "Not enough history yet — add ~5 shows you've watched or rated to generate recommendations." with a link back to Watch History.
- **Filtered rec list (zero matches):** "No recommendations match these filters." with a "Clear filters" action.
- **Co-watch list — asymmetric signal:** Co-watch is generated whenever *either* user has sufficient signal (the LLM works with whatever overlap exists; weak Co-watch is preferable to no Co-watch). When only one user has substantial signal, a small inline note at the top reads: *"Co-watch gets stronger as [partner name] adds more shows."* Recs are still rendered. Co-watch only shows the standard "no history yet" empty state when **both** users have insufficient signal.

#### 6.6.2 Loading States
- **Operations expected to take >200ms:** show a skeleton in the shape of the eventual content.
  - Watch History page first load: skeleton rows.
  - Show Detail open: skeleton card.
  - In-Progress view first load: skeleton entries.
- **Operations <200ms** (in-memory filtering, cached data): no indicator (avoids spinner flicker).
- **TMDb autocomplete:** 200ms debounce on keystroke + small inline spinner in the input while a request is in flight.
- **Rec generation has its own pattern** — see §6.4.7 (background gen + nav pill + skeleton cards + dimmed stale list).

#### 6.6.3 Error States
- **Pattern: inline error with Retry, never block the whole screen.**
- **TMDb autocomplete failure:** "Search unavailable — try again" shown inline under the input; the search input itself remains usable.
- **Show Detail fetch failure:** error card in place of the data with a Retry button; the drawer / route stays open.
- **Provider data missing for a recommendation card:** the platform-chip area shows "Availability unknown" instead of platforms; the rest of the card renders normally.
- **Rec-generation errors:** see §6.4.7 latency/error subsection.
- The user is never left stranded with a wiped-out screen; every error offers a path forward.

#### 6.6.4 Partial-Data States
- **Pattern: hide missing fields silently; show only what we have.** Missing data should not generate visible noise.
- **No trailer available:** hide the trailer link; no "Trailer: not available" placeholder.
- **No poster available:** render a default poster (a genre-colored block with the title text overlaid).
- **Provider data missing:** show an "Availability unknown" chip in place of platform chips.
- **TMDb production status null:** hide the production-status line in the In-Progress entry.
- **Total episode count null on an ongoing show:** show "Season X, ongoing" without the episodes-remaining number.

## 7. Non-Functional Requirements

- **Performance:** Target rec generation in ~15s. UI shows a "taking longer than usual" note at 30s and hard-times-out with a Retry option at 60s (one silent auto-retry on transient errors). See §6.4.7 for the full latency/error UX. Watch history pages should load in <1s on cached data.
- **Reliability:** Single-region deployment is fine. Daily DB backup.
- **Security:** API keys (TMDb, Anthropic) stored server-side only, never sent to the client. Auth tokens stored in HTTP-only cookies.
- **Cost:** Should run for well under $20/month at two-user scale. Claude API costs are the largest variable.
- **Privacy:** No third-party analytics. User data stays in the project's own database.

### 7.1 Accessibility (v1 floor)

This is the realistic accessibility floor for a 2-user personal app — not a full WCAG 2.2 AA audit, but enough that the app is usable by keyboard, screen readers, and users who can't rely on color alone (which also future-proofs against dark mode).

- **Keyboard:**
  - All interactions reachable without a mouse.
  - Vote pills are `<button>` elements (not `<div>`); arrow keys navigate within a vote group.
  - Esc closes drawers / modals / inline-expanded card states.
  - Tab order follows visual order.
- **Focus management:**
  - Visible focus ring on all interactive elements (use `focus-visible:` Tailwind utilities).
  - After casting a vote, focus stays on the pill (no jump).
  - Opening a Show Detail drawer / modal moves focus inside; closing returns focus to the originating card or list item.
- **ARIA / screen readers:**
  - `aria-label` on icon-only controls (e.g., "Agree — your vote", "Jaimie voted Disagree").
  - `aria-pressed` on toggle pills (vote, filter chips).
  - Form inputs have associated `<label>` elements, not just placeholders.
- **Color & contrast:**
  - Minimum 4.5:1 contrast for text, 3:1 for UI components (Tailwind's lightest grays often fail this — pick deliberately).
  - **State is never conveyed by color alone.** Vote state, status, availability — always pair color with an icon or text label so the meaning survives dark mode, color-vision differences, and printing.
- **Motion:**
  - Respect `prefers-reduced-motion` for the rec-gen shimmer, skeleton animations, card transitions, and drawer slides. Provide an instant alternative state when reduced motion is preferred.
- **Not in v1 scope:**
  - Full WCAG 2.2 AA audit.
  - Screen-reader testing across multiple AT (NVDA / JAWS / VoiceOver) matrix.
  - Internationalization or right-to-left support.

### 7.2 Responsive Design

The app is responsive web only (no native mobile per §3). Two breakpoint sets matter: **mobile ≤768px** and **desktop >768px**. The nav and rec card layouts diverge between them.

- **Primary navigation:**
  - **Mobile (≤768px):** The three rec lists (Co-watch / Corey / Jaimie) are the **primary navigation via a bottom tab bar**. Watch History, Search, and Settings live in a secondary top-right menu (hamburger or overflow). The identity chip (§6.1) is persistently visible in the top header. *This structure assumes M3+; pre-M3 builds will surface History/Search as primary since the rec tabs have no content yet.*
  - **Desktop (>768px):** A top nav contains Watch History / Recommendations / Search / Settings; within Recommendations, Co-watch / Corey / Jaimie are top tabs. Identity chip lives in the top-right corner.
- **Recommendation card on mobile (compact view, 375px width):** **Poster-first hero layout.** Large poster fills most of the card width; title sits below or overlaid; vote pills and a "More" affordance sit at the bottom. Platform chips, partner-vote indicator, LLM explanation, and the "Add to Want to Watch" button are **only visible in the expanded state**. Trade-off: visually striking, but most voting decisions require expanding the card first.
- **Recommendation card on desktop:** standard compact layout per §6.4.3 (poster + title + 2 chips + 1-line explanation + vote pills + partner-vote indicator + "More" + "Add to WTW" all visible without expanding).
- **Filters on mobile:** a "Filters" button in the header opens a **bottom sheet** with platform / genre / status / whose-pick chips. Filters do not occupy persistent screen space on mobile.
- **Filters on desktop:** persistent filter rail (left or right) on the recommendations view.
- **Touch targets:** minimum 44 × 44px for all interactive controls on mobile (matches §7.1 ARIA / focus requirements).

### 7.3 Theming

- **Light and dark themes are both shipped in v1**, designed together from day one.
- **Default to `prefers-color-scheme`** — the user's OS preference selects the theme automatically.
- A theme override toggle in Settings lets the user force Light, Dark, or System.
- All color tokens are designed against both modes simultaneously; no color decision is made in isolation against a single mode.
- Vote-state colors, status colors, the "Unavailable on your subscriptions" badge, and other state-bearing UI all have parallel light and dark definitions that meet §7.1 contrast requirements (4.5:1 text / 3:1 UI components) and never convey state by color alone.
- Rationale: couples watch TV in dim rooms; defaulting to bright white is the wrong call for this context. Retrofitting dark mode after shipping light-only is expensive (every color decision gets revisited).

## 8. User Experience / Key Flows

### Flow A — First-time setup
1. User enters their passcode at the login screen.
2. Selects which streaming subscriptions they have.
3. Sees empty dashboard with a prompt to search for and add shows.
4. After adding ~5 shows, recommendation lists populate.

### Flow B — Adding a show
1. User clicks "Add Show," types a title.
2. TMDb autocomplete shows matches with year and poster.
3. User picks a match and chooses a status (`Want to Watch`, `Watching`, `Paused`, `Completed`, or `Dropped`). If `Watching` or `Paused`, sets `current_season`. Optionally rates Like / Dislike / Meh — available on any status (§6.3).
4. Show appears in the appropriate list.

### Flow C — Browsing recommendations
1. User opens the app and lands on **Co-watch** by default (the most distinctive feature of the app and the daily-use case for the household). The Recommendations view has three peer tabs: Co-watch, Corey, Jaimie.
2. Each list is ranked; each card shows the show's metadata, vote pills, and (in M4+) the partner's vote. Rank is implicit from card position (§6.4.3).
3. Filters live in a persistent filter rail on desktop or in a bottom sheet behind a "Filters" button on mobile (§7.2): platform, genre, status, whose pick.
4. User casts a vote; it is recorded immediately. The current list does not re-rank — the vote feeds the next manual or auto-triggered Refresh (§6.4.4). Partner-vote visibility ships in M4.
5. User can tap any card's poster or title to open Show Detail (drawer in-app, `/show/[tmdb_id]` route for shared links — see §6.6) with trailer, full metadata, providers, and quick actions to change status (Want to Watch / Watching / Paused / Completed / Dropped) and rating.
6. A single Refresh button regenerates all three rec lists in parallel. Generation runs in the background; the user can navigate freely while a small pill in the nav indicates the refresh is in flight. See §6.4.7 for the full loading-state UX. Subscription changes in Settings also auto-trigger a regeneration.

## 9. Technical Architecture (proposed)

- **Frontend & backend:** Next.js 15 (App Router) + TypeScript, deployed to Vercel.
- **Database:** SQLite via Prisma (local file) for v1; migrate to Postgres if needed.
- **External APIs:**
  - **TMDb** — metadata, community ratings, trailers, Canadian watch providers (free tier).
  - **Anthropic Claude API** — recommendation reasoning. Use Claude Sonnet 4.6 as default; consider prompt caching on the watch-history system prompt.
- **Auth:** simple passcode middleware (per-user hashed passcode, HTTP-only cookie session). No NextAuth, no email provider.
- **Styling:** Tailwind CSS.

### Data model (sketch)

- `User` — id, email, display_name (`Corey` | `Jaimie`)
- `UserSubscription` — id, user_id, platform_key (e.g., `netflix`, `disney_plus`, `apple_tv_plus`, `crave`, `prime_video`)
- `Show` — id, tmdb_id, title, poster_url, genres[], total_seasons, total_episodes, tmdb_rating, trailer_url, last_synced_at
- `ShowProvider` — id, show_id, platform_key, region (`CA`), monetization_type (`flatrate` | `ads` | `free`)
- `WatchEntry` — id, user_id, show_id, status (`want_to_watch` | `watching` | `paused` | `completed` | `dropped`), current_season, user_rating (`like` | `dislike` | `meh` | null), notes, updated_at
- `RecommendationRun` — id, list_type (`global` | `corey` | `jaimie`), generated_at, input_tokens, output_tokens, model (the Claude model used, e.g., `claude-sonnet-4-6`), cost_usd (computed from token counts × model pricing at run time). Powers the cost-tracking success metric in §10 and the M5 cost-monitoring alert.
- `RecommendationItem` — id, run_id, show_id, rank, llm_explanation_short (≤100 chars, shown on compact rec card), llm_explanation_long (≤300 chars, shown on expanded rec card). Both are generated by the LLM in the same run; the prompt explicitly asks for both versions per recommendation.
- `RecommendationVote` — id, item_id, user_id, vote (`agree` | `disagree` | `maybe`), voted_at. **Unique on (item_id, user_id)** — re-voting overwrites the existing row; no vote history is retained in v1.
- `ContinuationDetectionLog` — id, show_id, detected_at (when the targeted in-progress refresh job first noticed new aired episodes beyond any user's `current_season`), tmdb_new_episode_air_date (TMDb's air date of the triggering episode), first_surfaced_at (when this detection first appeared as a continuation in a `RecommendationRun`, nullable until surfaced). Powers the §10 "continuations within one week" success metric, computed as `first_surfaced_at - tmdb_new_episode_air_date ≤ 7 days`.

## 10. Success Metrics

(For a personal-scale app these are more "did we build the right thing" checks than analytics.)

- Both users add at least 20 shows within the first two weeks of use.
- Each user votes Agree (or adds to Want to Watch) on at least **1 in 5 of the top 10 ranked items per rec run**, on average. (Computable from rank without per-card impression tracking — no `RecommendationImpression` table needed in v1.)
- Continuations correctly identify new seasons within one week of TMDb listing them. Computed from `ContinuationDetectionLog.first_surfaced_at − tmdb_new_episode_air_date ≤ 7 days` (see §9).
- Recommendations exclude unwatchable shows **100% of the time**, except for shows whose provider availability changed within the 14-day provider-cache refresh window (§6.5). Any other miss is a bug, not an accepted error rate.
- **Monthly Claude API spend stays under $15** (well below the §7 $20 budget). Implemented in M3c Phase 22: every Anthropic call writes a row to `LlmCallLog` with `inputTokens`, `outputTokens`, and a `costUsd` computed from per-MTok rates (Haiku $1/$5, Sonnet $3/$15). `getBudgetStatus()` sums the current calendar-month spend and returns one of three states:
  - `ok` — under 75% of the cap.
  - `warning` — between 75% and 100%; the /settings BudgetStatusCard surfaces a yellow indicator.
  - `exceeded` — at or past 100%; `generateRecommendations` short-circuits with `error: "budget_exceeded"` before calling Anthropic, the /recs failure pill folds the message into the refresh UI ("Monthly Anthropic budget hit — paused until next month"), and the /settings card switches to the danger state. Spend resets at the start of the next UTC month.

## 11. Risks & Future Considerations

### Risks
- **LLM hallucination:** Claude could recommend shows that don't exist or misattribute seasons. *Mitigation:* require each recommendation to include a TMDb ID and validate against TMDb before showing; provider data is also fetched at validation time per §13 M3.
- **API cost drift:** Heavy recommendation use could push Claude API costs up. *Mitigation:* manual-Refresh + sub-change-only auto-trigger (§6.4.7) keeps run frequency bounded; per-run token logging + 75%-of-budget alert (§10, §13 M5).
- **TMDb rate limits:** Free tier is generous but not unlimited. *Mitigation:* aggressive local caching (§6.5).
- **Provider data freshness:** Streaming availability in Canada changes monthly; stale cache could mislead. *Mitigation:* refresh providers data per show every 14 days (§6.5); §10 metric explicitly carves out this window.

### Future considerations (deferred from v1)
- **Smarter Disagree learning.** v1 only excludes the specific disagreed show from future runs. A later iteration could pass Disagrees into the LLM prompt as "avoid similar to" context, or — once enough vote history exists — cluster disagreed shows and adjust genre/tone weighting in rec generation. Deferred because over-correcting from a tiny 2-user vote pool would likely produce worse recs early on; revisit after a few months of real usage.
- **Inactivity-driven progress nudges.** Periodically check `Watching` entries that have not been updated recently (e.g., 60 days) and where TMDb shows additional aired episodes; surface an inline "Still on Season X?" prompt in the In-Progress view with quick actions (advance, mark as Paused, mark as Completed). Deferred from v1 to keep the In-Progress view quiet until real usage patterns reveal whether this is needed.
- **Streamed rec generation for perceived latency.** Use Anthropic's streaming API with structured output (e.g., JSON Lines per recommendation) so cards populate one-at-a-time as the LLM emits them. Reduces perceived wait from ~15s blank to ~2s first-card. Deferred from v1 because it requires a streaming parser and partial-state rendering on top of the batch implementation; revisit once batch generation is stable end-to-end.
- **CSV bulk import of watch history.** Originally proposed in v1 as a bootstrap convenience for users with hundreds of pre-app shows watched. Deferred because the manual search-and-add flow may be sufficient in practice for a 2-user household; revisit after M1–M3 are in use and we see whether bootstrap friction is real. If shipped, design considerations: canonical CSV format (header row, strict status enum `want_to_watch` | `watching` | `paused` | `completed` | `dropped`, RFC 4180 quoting, optional `rating` / `season` / `notes` columns); upload → parse preview → auto-match against TMDb → reconciliation screen for ambiguous rows (per-row Accept / Pick different / Search manually / Skip) → commit; user confirms all matches before writing (no silent auto-commit on a 2-user app where mismatches pollute personal taste data); desktop-only experience with a friendly mobile fallback message, since bulk import is a one-time setup operation.

## 13. Milestones

Each milestone is intended to be testable end-to-end on its own — M3 ships a working rec feedback loop (per-user voting), not just static rec lists, so quality can actually be steered.

1. **M1 — Foundations**
   - Project scaffold; Next.js + TypeScript + Prisma + SQLite + Tailwind on Vercel.
   - Passcode auth for both users (HTTP-only cookie session).
   - User settings: active subscriptions (editable).
   - TMDb autocomplete search (poster + year disambiguation, **no full detail view yet** — that ships in M3).
   - Watch entry add/edit/delete with all five statuses (`Want to Watch`, `Watching`, `Paused`, `Completed`, `Dropped`).
   - Like / Dislike / Meh ratings settable on any status.

2. **M2 — In-Progress**
   - In-Progress view: episodes-remaining, TMDb production status display (with "may change" caveat), "Unavailable on your subscriptions" badge.
   - Inline `+1 / -1 season` controls and "Finished it" shortcut on Watching entries.
   - Targeted metadata refresh job for in-progress shows (new-episode detection) — required for M3 continuations.

3. **M3 — Recommendations v1 + basic voting**
   - Three ranked lists (Co-watch, Corey, Jaimie); LLM batch rec generation.
   - Background generation: nav pill, dimmed stale list, skeleton cards.
   - Latency UX: 30s notice, 60s timeout + Retry, one silent auto-retry on transient errors; mood input persists across failures.
   - Show detail view (trailer, providers, full metadata).
   - Filters (platform / genre / status / whose-pick).
   - Subscription-change auto-refresh trigger + client-side stale-list filter.
   - Provider data fetched + cached at rec-validation time for every recommended TMDb ID.
   - **Basic Agree / Disagree / Maybe voting (per-user only, no partner visibility).**
   - **Votes feed next refresh — closes the feedback loop.**
   - "Add to Want to Watch" button on rec card (separate from vote pills).
   - Claude / TMDb error handling on the rec-gen path.

4. **M4 — Voting loop + partner visibility**
   - Partner-vote visibility on every card (e.g., "Jaimie: Agree, Corey: Maybe").
   - Co-watch split rule (Agree + Disagree → demote next run).
   - Disagree-on-continuation prompt flow (move to Paused / Dropped).
   - Refined LLM context using accumulated vote history.
   - Like / Dislike / Meh feedback-loop quality polish.

5. **M5 — Polish**
   - Provider / availability refresh job (14-day cadence).
   - Show metadata + ratings weekly refresh job.
   - Accessibility audit against §7.1.
   - Responsive polish; dark mode (per §7.2 if chosen).
   - Claude API cost monitoring + alert at 75% of monthly budget.
   - Deployment hardening, DB backup automation.
